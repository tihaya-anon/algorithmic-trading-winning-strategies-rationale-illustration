# Assertion-Evidence Approach Reference

Use this reference when designing, revising, or critiquing technical and educational presentations with the assertion-evidence approach (AEA).

## Core Model

AEA replaces the common topic-plus-bullets slide with two coordinated parts:

- Assertion headline: a concise, complete sentence that states the slide's main message.
- Visual evidence: a graph, image, diagram, equation, table, comparison, annotated screenshot, or structured text layout that supports the assertion.

The approach is most useful for scientific, engineering, technical, analytical, course, and research presentations where the audience needs to understand relationships, mechanisms, results, tradeoffs, and evidence.

## Research-Grounded Rationale

Use AEA because it changes both audience processing and presenter preparation:

- Sentence headlines orient the audience to the point of the slide instead of merely naming a topic.
- Visual evidence supports dual verbal and visual processing, improving the chance that audiences understand and remember technical material.
- Replacing bullet lists with visual evidence reduces split attention between reading dense text and listening to the presenter.
- Preparing assertion-evidence slides forces presenters to decide what each slide proves, which can deepen the presenter's understanding of the material.
- Decorative visuals are not evidence; use images and graphics that represent the argument, data, mechanism, object, process, or comparison.

## Slide-Level Rules

Apply these rules to every content slide unless the user's format constraints require an exception:

- Write the headline as a full-sentence assertion, not a noun phrase.
- Keep the headline to one or two lines.
- Make the headline specific enough that a reader can infer why the slide exists.
- Support the headline with visual evidence in the body.
- Prefer one dominant evidence object over several unrelated objects.
- Use bullets only as a last resort; when text lists are necessary, convert them into a table, process diagram, labeled comparison, or compact callout structure.
- Place secondary explanation in speaker notes, not in the slide body.
- Use whitespace to separate evidence regions and preserve visual hierarchy.
- Keep labels large enough to read from the intended viewing distance.
- Avoid all-caps body text and decorative template art.

## Deck-Level Rules

Plan the talk as a sequence of assertions:

1. Draft the talk's core message and audience decision or learning outcome.
2. Create a sentence-outline where each slide headline is one assertion in the story.
3. Check that adjacent assertions create a coherent argument, not a topic inventory.
4. Select evidence for each assertion before polishing visual style.
5. Use title, mapping, transition, and conclusion slides to orient the audience.
6. End with the strongest conclusion slide and keep it visible during questions when appropriate.

## Evidence Patterns

Choose evidence based on what the assertion claims:

- Trend or change: line chart, slope annotation, before/after plot, event markers.
- Comparison: aligned bar chart, difference plot, table with emphasized deltas, paired images.
- Mechanism or process: flow diagram, causal chain, system schematic, annotated architecture.
- Spatial or physical object: photograph, map, labeled drawing, cross-section, microscopy or instrument image.
- Ranking or prioritization: sorted bar chart, quadrant, matrix, decision table.
- Risk or uncertainty: interval plot, fan chart, scenario table, sensitivity chart.
- Definition or framework: minimal diagram, taxonomy, concept map, worked example.
- Code or math: highlighted snippet, equation with callouts, transformation steps, input-output example.

## Revision Checklist

Use this checklist when converting existing slides:

- Convert phrase titles into assertion headlines.
- Identify the one claim each slide needs to prove.
- Delete bullets that duplicate narration.
- Move operational details into speaker notes or handouts.
- Replace bullets with evidence or a structured visual relationship.
- Highlight only the evidence needed for the assertion.
- Remove decorative images that do not support the assertion.
- Add annotations that explain what to look at, not paragraphs that explain everything.
- Verify that all text remains readable on the smallest expected display.
- Check that the deck tells a sequence of claims with a beginning, middle, and ending.

## Common Transformations

Use these transformations during slide editing:

- "Backtesting Results" -> "The strategy's alpha disappears after transaction costs."
- "Data Cleaning" -> "Survivorship and look-ahead filters remove most false signal strength."
- "Model Architecture" -> "A two-stage model separates regime detection from position sizing."
- "Limitations" -> "The largest remaining risk is liquidity decay under crowded exits."
- Five bullets describing causes -> causal diagram with labeled arrows.
- Bulleted parameter list -> compact table grouped by decision purpose.
- Long quotation -> abridged quotation or paraphrased takeaway with source in notes.

## Speaker Notes Guidance

Use speaker notes to carry material that bullets usually carry:

- Include the spoken explanation of visual evidence.
- Keep caveats, citations, method details, and backup numbers in notes.
- Avoid writing a script that must be read verbatim.
- Prefer short speaking prompts and evidence interpretation cues.
