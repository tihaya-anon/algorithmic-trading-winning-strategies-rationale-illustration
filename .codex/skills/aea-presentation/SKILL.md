# AEA Presentation

## Overview

Create and revise presentations around the Assertion-Evidence Approach (AEA): use a complete-sentence assertion as each slide headline, then support that assertion with visual evidence. Favor clear claims, specific evidence, and speaker notes over topic labels, dense bullet lists, and decorative visuals.

Load `references/aea-method.md` when a task requires detailed AEA rationale, a full slide conversion checklist, source-backed explanation, or a critique of an existing deck.

## Trigger Conditions

Use this skill when the user asks to:

- Create slides, Quarto/revealjs slides, PowerPoint content, speaker notes, or a talk outline using AEA.
- Convert existing bullet-heavy slides into assertion-evidence slides.
- Review a presentation for clarity, evidence quality, or technical communication quality.
- Improve scientific, engineering, analytical, course, financial, or research slides.
- Produce slide titles, headlines, layouts, visual evidence ideas, or narration in an assertion-evidence style.

## Core Rules

Apply these rules by default:

- Write each content-slide headline as a concise sentence that states the slide's main assertion.
- Support the assertion with visual evidence: graph, diagram, image, table, equation, annotated screenshot, structured comparison, or another evidence-bearing visual.
- Remove bullets that duplicate narration; move supporting detail into speaker notes or a handout.
- Use decorative imagery only when it directly orients the audience to the subject.
- Keep one main claim per slide.
- Keep slide text short, readable, and visually subordinate to the evidence.
- Make the slide body show relationships, not just list items.

## Workflow

1. Clarify the presentation goal, audience, setting, duration, and required format.
2. Draft the deck as a sequence of sentence assertions before writing slide bodies.
3. For each assertion, select the strongest available evidence pattern.
4. Design each slide around one dominant evidence object with clear labels and callouts.
5. Move caveats, methods, citations, and narration prompts into speaker notes.
6. Check that the assertion sequence forms a coherent story.
7. Audit the final deck for readable type, low clutter, evidence-message alignment, and minimal decorative content.

## Creating New Slides

For each slide, produce:

- `headline`: one complete-sentence assertion.
- `evidence`: the visual proof or demonstration to place in the slide body.
- `layout`: a practical arrangement of headline, visual evidence, labels, and callouts.
- `speaker_notes`: concise prompts for explaining the evidence without reading the slide.
- `backup_detail`: optional methods, caveats, citations, or data notes that belong outside the visible slide.

Prefer fewer, stronger slides when the evidence needs time to explain. Avoid creating one slide for every topic in the source material.

## Revising Existing Slides

Convert slides with this sequence:

1. Identify the real claim the slide needs the audience to remember.
2. Rewrite the title as that claim.
3. Delete or move bullets that are speaker notes rather than audience evidence.
4. Replace bullet lists with charts, diagrams, tables, annotated images, or structured text relationships.
5. Highlight the evidence that directly supports the assertion.
6. Preserve necessary caveats in notes or backup slides.

## Reviewing Slides

Evaluate each slide with these questions:

- Does the headline state a complete, specific assertion?
- Does the body provide evidence for that assertion?
- Could the audience understand the main point without hearing every spoken word?
- Does any visible text compete with the speaker's narration?
- Are graphics representative and explanatory rather than decorative?
- Does the slide sequence build an argument rather than a topic list?

Report issues as concrete rewrites and layout changes, not abstract presentation advice.
